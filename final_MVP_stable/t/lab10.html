Lab

TETRIS!

The game selected for this year's lab 10 is tetris. You will implement a simplified form of the game for this lab, but you are free to do extra work for some extension credit.

This is a good place to look to find information about pieces and scoring. The requirements are listed below.

You may complete this lab as an individual or as a team of 2.

Basic Game

To receive credit for this lab, you must design and implement the following elements of the game:
  • (7 points) Create at least two different pieces (block configurations). Each piece should have a color associated with it.
    Note: if one of the pieces you choose is the one with 4 blocks in a line, it must be in a vertical column in your implementation.
  • (8 points) A score panel showing:
    • Current score - Scoring does not have to adhere to Tetris rules of scoring but there must be a score per complete row.
    • Next piece descending - Should be a picture not text
  • (15 points) One piece descends at a time, a new piece appears when the current piece hits another piece or the bottom
  • (3 points) Current piece is chosen at random
  • (15 points) Pieces respond to key clicks that move them either left or right (rotation is not required)
  • (10 points) Completed rows are removed and scored. A row is considered complete if it is filled with blocks with no spaces.
  • (10 points) Pieces move down when a row(s) below is removed.
  • (2 points) Cause your game to end when either the game display fills up or the player reaches some time or score threshold.
  • Code requirement:
    • Graphical Objects you implement should implement the Anim interface.
    • You are welcome to extend Anim for your own purposes.

Grading

Lab 10 will be graded out of 100 points. Points are alloted as follows:
  • 2 points Studio (4/7)
  • 13 points Code Review:
    Deadline for the code review is Thursday, 4/21. If you plan on doing this on the last day you will be asked to sign up for a slot in advance for that day. This part may be demoed during lab times, TA office hours, or instructor office hours.
    • You don't have to have everything done to have a code review but you should have a significant portion of your design and implementation done. Present the overview of your code structure. It's okay if you haven't fully implemented all specifications.
    • Points are allocated as follows
      • (1 point)List all classes
      • (5 points)For each class have a file and in comments at the top of the file specify the following:
        • has-a's (what information/state will the instance variables represent)
        • functionality, how will you access and manipulate the has-a's (what methods will you have)
      • (4 points)Explain the relationship between the classes, how they are used to implement the game
      • (3 points)This is a reasonable design for the problem
      • (-10 points)If you cannot demonstrate that this is your original design and all team members have participated
  • 70 points Completion of Basic Game requirements. Points will be allotted for each feature successfully implemented as specified in the Basic Game section above.
    You will need to sign up in advance for a slot on 4/26 or 4/28 to demo your game. A form is in the works for you to do this. You will not be able to demo this part at TA office hours.
  • 10 points code modularity and generalization
    • Did you implement and design reusable, self-contained pieces of code as Objects and Methods?
    • If you used some piece of code over and over did you put it in a method?
    • Did you use Array, Lists and Sets where appropriate instead of creating a lot of individual variables?
    • The majority of the functionality of your code should be in your Objects not in your main method.
  • 5 points cosmetic style: indentation and comments
  • -25 points Used a concept outside of the scope of the class and could not explain it.

How do I process keyboard input?

Please use the following keys for your game,
Key Meaning
A or a Left
D or d Right

The methods of use to you from Sedgewick's API:

boolean StdDraw.hasNextKeyType()
You can call this method to see if the player has typed any key. It does not wait for input.
char StdDraw.nextKeyTyped()
This method returns the next key typed. You should only call this method if the above method has returned true.

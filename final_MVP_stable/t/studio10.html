Tetris

This week's studio is meant to get you used to thinking about how you might implement Tetris. You are not obligated to use any of the code you develop today in your final implementation. The group you work with today does not have to include the person you plan to collaborate with for the final project.

In the videos for this week, you learned about the Anim interface , which insists that objects can draw themselves and say whether they are done. The graphical objects that you create for your version of Tetris should implement this interface. You also reviewed the standard animation loop that we use for all animation. In this studio, you will start to think about how to break Tetris down into codable pieces. Your work will include identifying objects that could be used in your implementation, specifying the has-a's and behaviors for one of these objects, and finally you will try creating and animating a block object. You will NOT receive as many step by step instructions so as to get you ready for working on the lab. However, TAs and the instructor will be around to provide help should you need it, but as much as possible draw on past labs and studios to try and figure things out.

Here are the steps:
  1. Gather your group around one workstation. Plan on taking turns at the keyboard.
  2. You should have familiarized yourself with Tetris by now.
  3. First try and come up with 3 elements of Tetris that you might represent as object classes in your implementation of the game. You are not obligated to use these classes in your final game so don't worry too much about it. This is just to get you used to thinking about how you might break the game into codable objects. As a start, you might think about the different graphical elements of the game or the different functions represented.
  4. Once you have 3 possible object types. Choose one of those types and write a specification of it. List what you think the has-a's are and the behaviors. You can do this on a piece of paper or create a class in the studio10 package of the studios folder and put it in the comments.
  5. DO NOT MOVE ON until you have shown your list of potential objects and your class specification to your TA so we can be sure you understand how to specify an object and to look for codable elements in a design problem.
  6. Code developed during lecture is in your studio workspace. Look in the studio10 package of the studios source folder.
    The code you should probably study is the VertBall class, since it implements Anim and moves in a vertical direction. You should also look at the GameOnVideo class, since it implements a standard animation loop using a list of objects implementing Anim much like you will do in this studio.

    This code is in your studio10 package.

  7. Examine the provided Anim interface.
  8. You are going to create a new class called VertBlock which implements the Anim abstract interface. You learned how to create a new class that inherits from an abstract interface in studio9. VertBlock represents a square block that is given:

    • a random color,
    • start position,
    • end position
    • and speed of descent.

    Each time it is drawn it advances down the screen as determined by the speed. It is first drawn at the start position and stops at the end position. The block should remain on the screen once it reaches the end position.
  9. DescendingBlocks which you can find in your studio10 folder contains the animation loop you will use for this studio. Blocks that will appear in your display should be added here.
  10. Using the description of VertBlock above and looking at the code for VertBall as a reference implement the class VertBlock and animate it using the DescendingBlocks loop. If you are confused how to do that, take a look at the GameOnVideo class for an example.
    Make sure to comment your code. If you are having difficulty implementing VertBlock, it might be a good exercise to write the specification of the class in terms of has-as and behaviors in the comments before writing any code. Strive for the cleanest, correct implementation.
  11. If you have time, add another block to the display after the first block has reached the bottom of the display. The second block should also start at the top and descend to the bottom. It should stop descending when it either reaches the bottom of the display or hits the first block. You may have to alter your VertBlock class to do this.
